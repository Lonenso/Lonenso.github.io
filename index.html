MySQL必知必会笔记

起因是面试的时候被问到LEFT JOIN, RIGHT JOIN. 之前在工作中没用过, 也不是很了解.
只感觉是集合的交并差运算.

买了<<MySQL必知必会>>过来看. 过一遍后记录一下.

仅记录以前不懂的, 不知道的, 先后顺序同原书籍, 并重新思考在之前的工作中可以怎么样更好的完成任务, 思考技术出现的原因, 不专注于技术本身.

汇总

  • Limit 语法, Limit 3, 4的语义并不清晰, 直接. 由于这个原因, MySQL 5 支持Limit的另一种语法, LIMIT 4 OFFSET 3
  • 除了”%”以外, 还有一个通配符”_”, 用来匹配单个字符.
  • 经常需要汇总数据而不需要把所有记录都检索出来, 为此提供专门的函数. 返回数据再进行处理是对时间/处理资源和带宽的浪费.
  • 表的联结, MySQL无OUTER JOIN
  • 在多次对同一张表进行子查询时, 可以通过别名进行自联结. 在有需要的情况下, 比较两种语句的性能.
  • 组合查询, 将同样列不同语句的查询结果拼接在一起.
  • 全文本搜索, MyISAM引擎支持, 性能好, 但不支持事务. InnoDB支持事务但不支持全文本搜索. 外键跨引擎可能不行.
  • 启用全文本搜索支持, FULLTEXT索引, 通过Match指定列, Against匹配搜索表达式, 同时提到一个技巧导入数据时不启用索引而是导入完成后再建立索引. 来减少花费的时间.
  • 提到全文本搜索有两种扩展模式, 一是查询扩展, 给我的感觉类似grep -C, 是以匹配行中包含的其他词来扩展查询. 二是布尔查询.
  • 索引全文本数据时, 短词(长度短, 可配置), 停用词, 非用词(出现在50%的行中, 匹配返回没有意义)会被忽略.
  • 提到通常数据库检索比存储数据重要, 根据场景, 可以调低动作的优先级. INSERT LOW_PRIORITY INTO
  • INSERT SELECT 不关心列名, 关心列的顺序.
  • 视图是一组SQL的集合(Dashboard), 重用SQL语句, 简化操作.
  • 存储过程, 批处理, 支持传参, 返回结果. 简化操作, 提供强大而又灵活的功能. (提到存储过程比使用单独的SQL语句性能更好)
  • 游标, 检索出数据后会需要对数据进行浏览, 跳转.
  • 事务处理, 提到一个之前不知道的savepoint, 指的是100个操作, 第80个操作错了, 可能没必要全部回滚, 而只需回滚到某一个操作即可.
  • 字符集(character set), 编码, 校对(collation), collation用来规定字符如何比较. 现在才知道 utf8_general_ci(case intensive), cs => case sensitive.
  • 备份数据, 提到为了保证所有数据被写到磁盘(包括索引数据), 可能需要在进行备份前使用FLUSH TABLES(在想现在还需要吗? 会不会集成到mysqldump里面去了)

实操

可能还得整一个cheatsheet

如何获取当前MySQL的保留字(查看官网当前版本手册)
如何区分是view还是普通表?
show full table;
如何列出存储过程?(之前没用过存储过程, 虽然现在也不打算用)

如何列出表的索引?(之前只会在gui下查看, 部署完了就没看过)
show indexes from products;

之前工作的时候大概有这么一个查询语句, 两层嵌套子查询. 涉及到两张表(A, B), 其中B是关系表, 表明A表中记录的父子关系.
语句做的事情是: 给定父节点, 得到属于该节点儿子的节点数量, 并可能加入一些其他的条件如儿子节点的上一次心跳时间等. 

> use test;
> show tables;
+----------------+
| Tables_in_test |
+----------------+
| a              |
| b              |
+----------------+
2 rows in set (0.02 sec)
mysql> select * from a;
+----+------+--------+
| id | name | role   |
+----+------+--------+
|  1 | aaa  | parent |
|  2 | bbb  | parent |
|  3 | ccc  | child  |
|  4 | ddd  | child  |
|  5 | eee  | child  |
+----+------+--------+
5 rows in set (0.03 sec)

mysql> select * from b;
+----+-----------+----------+
| id | parent_id | child_id |
+----+-----------+----------+
|  1 |         1 |        3 |
|  2 |         1 |        4 |
|  3 |         2 |        5 |
+----+-----------+----------+
3 rows in set (0.04 sec)

即aaa是ccc和ddd的父节点, bbb是eee的父节点

原语句:

select count(1)
from a
where id IN (
    select child_id
    from b
    where parent_id = (
        select id
        from a
        where name = "aaa"
          and role = "parent")
);

现语句:

select count(1)
from a,
     a as a2,
     b
where a2.name = "aaa"
  and a2.role = "parent"
  and a.id = b.child_id
  and a2.id = b.parent_id;

目前来看我认为自联结比原来简洁, 但实际性能两者还需要比较一下.

后记

之前复制粘贴SQL语句查询的时候, 觉得枯燥和繁琐. 应该提取出视图和存储过程先过渡一下的. 那时候想的是整一个统计部署的页面, 也没时间就耽误了.
这下了解论坛上大家提到的存储过程很长, 业务逻辑在SQL层是什么情况了.
之前不知道数据库有游标, 只是之前java的作业里直接使用的MySQL的连接件操作游标, pymysql也操作过游标.
之后<<高性能MySQL优化>>再说

1453. Maximum Number of Darts Inside of a Circular Dartboard

leetcode某周周赛题. 感觉很有思路, 但一直WA. 后来搞明白要考虑斜率.

思路

一开始想的是三角形的外接圆. 打算用三点求圆. 发现好像不好和半径进行关联. 后来想想半径+两点就能画圆了. 于是步骤如下

  1. 两两坐标画圆, 根据两点坐标+圆的半径画圆
  2. 判断其他点是否在圆内(比较两点距离和半径的关系)
  3. 取最大值

画圆

假设两点为$A(x_1, y_1), B(x_2, y_2)$. 过中点D做垂直平分线. 如果半径r小于AD, 则不存在圆. 终止.
假设存在圆C.
过点B, 做x轴的平行线, 过点A做y轴的平行线. 两条线交点M. 过点D做y轴的平行线, 与直线BM交点N.
过圆心点C做垂直使得CO垂直DN.

此时$∠MAB = ∠NDB = ∠DCO = θ$

$cosθ = |y_1-y_2|/AB$
$sinθ = |x_1-x_2|/AB$

根据AB的斜率不同, 圆心的坐标不同

  • 当AB斜率小于0时
    $x_C$ = $x_D ± CD * cosθ$
    $y_C$ = $y_D ± CD * sinθ$

  • 当AB斜率大于0时
    $x_C$ = $x_D ∓ CD * cosθ$
    $y_C$ = $y_D ∓ CD * sinθ$

其实不需要考虑两边, 一直选择某一边即可.
但看了下别人的写法, 好像是相似三角形. 别人好像没考虑斜率, 但是代码看不懂..

LC_ALL=C 解决语言乱码

起因是开发的Go程序在不同的机器上运行, 而这些机器中有些编码不同(GB2312, GBK). 而Go使用的编码是UTF-8. 导致程序的输出是乱码.
使用LC_ALL=C, 临时修改编码

LC_ALL=C ./main

源码编译Openssl时遇到的疑难杂症

环境

  • centos 6.5
  • openssl 1.1.1d

安装步骤

#!/bin/bash
curl -O https://www.openssl.org/source/openssl-1.1.1d.tar.gz
tar -C /usr/local/src/ -xf openssl-1.1.1d.tar.gz
cd /usr/local/src/openssl-1.1.1d
./config --prefix=/usr/local/openssl --openssldir=/usr/local/openssl shared zlib
make 
make test
make install
mv /usr/bin/openssl /usr/bin/openssl.bak
mv /usr/include/openssl/ /usr/include/openssl.bak
ln -s /usr/local/openssl/bin/openssl /usr/bin/openssl
ln -s /usr/local/openssl/include/openssl/ /usr/include/openssl
echo "/usr/local/openssl/lib" >> /etc/ld.so.conf

openssl: error while loading shared libraries: libssl.so.1.1: cannot open shared object file: No such file or directory

这是由于openssl库的位置不正确导致的, 创建软链接解决. 

ln -s /usr/local/lib64/libssl.so.1.1 /usr/lib64/libssl.so.1.1
ln -s /usr/local/lib64/libcrypto.so.1.1 /usr/lib64/libcrypto.so.1.1

Can’t locate Test/Harness.pm in @INC

yum install perl-CPAN
perl -MCPAN -e shell

一路yes, 到cpan[1]

install Text::Template

直到安装完成, 退出, 再make test

note: 晚上加班安装的openssl, 比较着急, 没有仔细看openssl官网

疫情新年远程办公感受

这篇由疫情期间在家远程办公, 每天onenote的记录得来. 主要记一下远程办公的感受:

  1. 个人的变化
  2. 与其他程序员远程办公的对比
  3. 与其他职业的人的对比

个人的变化

家在南方沿海, 2月4日 左右无阳光, 冷得发抖. 加上远程办公, 网络连接真不能算流畅, 远程连接有几个IDE的快捷键不能用, 人处于一个比较烦躁的状态

第一次在家工作, 特别地, 家里也没有专门工作的房间, 在卧室里, 就总想睡, 想趴下. 没有办公的状态, 归结为心还没沉下来.

作息不同, 又可能要24 hours on call. 部门内作息时间交叉, 也不像在公司, 下班就下班了, 一开始还会码到睡觉前.

后来从我的onenote记录来看, 情况好得多, 我在每一天, 每一页的记录都增加了一串流水账, 这也是从别人那里学来的. 

这个笔记的顶部右边放作息
分别插入: 保证8小时即可.
早上上班, 早上下班
中午上班, 下午下班
晚上上班, 晚上下班

中间在细致到扣除吃饭, 睡觉, 忽然出门等等.

再加上, 这段时间内只专注于代码, 没有平常在公司碰到的杂活. 没有公司那么吵, 更加的安静和专注.

与其他程序员远程办公的对比

有的公司在疫情期间, 连远程办公都没法提供, 有的不需要上班, 有的心里一直想着去上班.

跟他们对比, 我有点幸运, 有点羡慕, 有点惭愧和不解.

我在的部门比较小, 人也不多, 办公的那段时间, 工作内容也比较直白. 论坛上其他人提到的交流不方便, 还没能深刻地体会到.
以及家里比较吵闹, 有小孩要照顾等等, 这些也没遇到, 相反, 疫情让父母呆在家里安心看电视, 追剧. 到村口帮忙疫情值班. 整体来说, 年后日子比较平静, 也比较少串门.
在家里也不用担心吃饭, 卫生的打扫. 父母也有个人使唤.

与其他职业的对比

有台电脑, 有远程连接就能办公. 这在我们村子那是件新奇的事情. 有的不发工资, 有的不知道什么时候上班, 有的降薪, 有的直接就开除了.

还有一个点是有些人由于事情太突然, 一下失去了现金流, 同时又有各种各样的贷款, 债务. 这个也是我还不能体会到的.
